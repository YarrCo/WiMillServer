# WiMill Server MVP

Это минимальный локальный сервер для MVP-проекта WiMill.

Сервер написан на `FastAPI`, запускается через `Uvicorn`, а данные хранит в локальной базе `SQLite`.

Документ написан для новичка: ниже есть простое объяснение, что делает проект, как его запустить и что означает каждая команда.

## Что умеет сервер

Сервер решает базовую задачу обмена данными между пользователем и устройством.

Схема работы такая:

1. Пользователь загружает файл на сервер.
2. Пользователь создает задачу для устройства.
3. Устройство периодически делает `poll` на сервер.
4. Если для устройства есть задача, сервер отдает ее устройству.
5. После выполнения устройство отправляет подтверждение.
6. Сервер сохраняет состояние устройства и задач в базе.

## Технологии

- `Python 3.12`
- `FastAPI`
- `Uvicorn`
- `SQLite`
- `Pydantic`

```text
WiMillServer/
├─ app/
│  ├─ main.py

│  ├─ models.py

│  ├─ uploads/

```

Что здесь находится:

- `app/main.py` - точка входа FastAPI-приложения.
- `app/database.py` - создание базы данных и каталогов хранения.
- `app/models.py` - Pydantic-модели запросов и ответов.
- `app/devices.py` - логика устройств: `poll` и список устройств.
- `app/jobs.py` - загрузка файлов, создание задач, завершение задач.
- `storage/uploads/` - сюда сохраняются загруженные файлы.
- `storage/devices/` - резервная папка под файлы устройств.
- `wimill.db` - локальная SQLite-база.

## Как установить зависимости

Команда:

```bash
pip install -r requirements.txt
```

Что она делает:

- `pip` - стандартный менеджер пакетов Python.
- `install` - говорит `pip`, что нужно установить зависимости.
- `-r requirements.txt` - взять список пакетов из файла `requirements.txt`.

После этой команды в систему или в виртуальное окружение будут установлены:

- `fastapi`
- `uvicorn`
- `pydantic`

## Как запустить сервер

Для локального запуска:

```bash
uvicorn app.main:app --reload
```

Что означает эта команда:

- `uvicorn` - ASGI-сервер, который запускает FastAPI-приложение.
- `app.main:app`
  `app.main` - Python-модуль `app/main.py`
  `app` - объект FastAPI внутри этого файла
- `--reload` - автоматический перезапуск сервера при изменении кода.

После запуска сервер будет доступен по адресу:

- `http://127.0.0.1:8000`
- документация Swagger: `http://127.0.0.1:8000/docs`

## Как запустить сервер для доступа из локальной сети

Если нужно открывать сервер с другого устройства в вашей локальной сети, запускайте так:

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

Что означает эта команда:

- `--host 0.0.0.0` - сервер слушает не только `localhost`, но и все сетевые интерфейсы компьютера.
- `--port 8000` - сервер работает на порту `8000`.
- `--reload` - сервер автоматически перезапускается при изменении файлов.

Если всё настроено правильно, доступ будет по адресу вида:

- `http://192.168.1.48:8000/docs`

Где `192.168.1.48` - IP-адрес вашего компьютера в локальной сети.

При первом запуске сервер сам создаст:

- файл базы данных `wimill.db`
- папку `storage/uploads/`
- папку `storage/devices/`

## Как работает база данных

Используется встроенная база `SQLite`, поэтому отдельный сервер базы данных не нужен.

В базе создаются 2 таблицы:

### Таблица `devices`

Хранит информацию об устройствах:

- `id` - внутренний числовой идентификатор
- `device_id` - строковый идентификатор устройства
- `device_name` - имя устройства
- `last_seen` - время последнего обращения к серверу
- `status` - текущее состояние устройства

### Таблица `jobs`

Хранит задания для устройств:

- `id` - внутренний числовой идентификатор
- `device_id` - для какого устройства создана задача
- `job_type` - тип задачи
- `file_name` - имя файла, связанного с задачей
- `status` - состояние задачи
- `created_at` - время создания задачи

### Возможные статусы задачи

- `pending` - задача создана, но устройство ее еще не забрало
- `running` - устройство уже получило задачу
- `done` - задача успешно выполнена
- `error` - при выполнении произошла ошибка

## API сервера

## 1. Проверка устройства и выдача задачи

Endpoint:

```text
POST /device/poll
```

Этот запрос устройство отправляет регулярно, чтобы:

- сообщить, что оно в сети
- обновить свой статус
- получить новую задачу, если она есть

Пример тела запроса:

```json
{
  "device_id": "mill01",
  "firmware": "0.1",
  "status": "idle"
}
```

Что происходит на сервере:

1. Обновляется `last_seen`.
2. Если устройство новое, оно добавляется в таблицу `devices`.
3. Проверяется очередь задач.
4. Если есть задача со статусом `pending`, она возвращается устройству.
5. Эта задача переводится в статус `running`.

Ответ, если задач нет:

```json
{
  "job": "none",
  "file_name": null
}
```

Ответ, если задача есть:

```json
{
  "job": "download",
  "file_name": "test.nc"
}
```

Пример команды `curl`:

```bash
curl -X POST "http://127.0.0.1:8000/device/poll" ^
  -H "Content-Type: application/json" ^
  -d "{\"device_id\":\"mill01\",\"firmware\":\"0.1\",\"status\":\"idle\"}"
```

Что означает эта команда:

- `curl` - утилита для отправки HTTP-запросов.
- `-X POST` - использовать метод `POST`.
- URL `http://127.0.0.1:8000/device/poll` - адрес endpoint.
- `-H "Content-Type: application/json"` - тело запроса отправляется в формате JSON.
- `-d "..."` - JSON-данные запроса.

## 2. Загрузка файла на сервер

Endpoint:

```text
POST /upload
```

Этот endpoint нужен, чтобы пользователь мог передать файл на сервер.

В текущем MVP файл отправляется как сырое содержимое запроса, а имя файла передается в строке запроса.

Пример:

```bash
curl -X POST "http://127.0.0.1:8000/upload?file_name=test.nc" ^
  -H "Content-Type: application/octet-stream" ^
  --data-binary "@test.nc"
```

Что означает эта команда:

- `-X POST` - отправить POST-запрос.
- `?file_name=test.nc` - имя файла, под которым он будет сохранен на сервере.
- `-H "Content-Type: application/octet-stream"` - передаем бинарные данные.
- `--data-binary "@test.nc"` - взять содержимое локального файла `test.nc` и отправить его как тело запроса.

Ответ:

```json
{
  "status": "ok",
  "file_name": "test.nc"
}
```

После этого файл появится в папке `storage/uploads/`.

## 3. Создание задачи для устройства

Endpoint:

```text
POST /jobs
```

Этот endpoint вызывает пользователь или внешний клиент, чтобы поставить устройству задачу на скачивание файла.

Пример тела запроса:

```json
{
  "device_id": "mill01",
  "file_name": "test.nc"
}
```

Что делает сервер:

1. Создает новую запись в таблице `jobs`.
2. Устанавливает `job_type = download`.
3. Устанавливает `status = pending`.

Пример команды:

```bash
curl -X POST "http://127.0.0.1:8000/jobs" ^
  -H "Content-Type: application/json" ^
  -d "{\"device_id\":\"mill01\",\"file_name\":\"test.nc\"}"
```

Что означает эта команда:

- `POST /jobs` - создать новую задачу.
- `device_id` - для какого устройства создается задача.
- `file_name` - какой файл устройство должно получить.

Ответ:

```json
{
  "status": "ok",
  "job_id": 1,
  "device_id": "mill01",
  "file_name": "test.nc"
}
```

## 4. Подтверждение выполнения задачи

Endpoint:

```text
POST /jobs/done
```

Этот запрос устройство отправляет после завершения задачи.

Пример тела запроса:

```json
{
  "device_id": "mill01",
  "file_name": "test.nc",
  "status": "done"
}
```

Возможные значения `status`:

- `done`
- `error`

Что делает сервер:

1. Находит последнюю подходящую задачу для устройства и файла.
2. Меняет ее статус на `done` или `error`.

Пример команды:

```bash
curl -X POST "http://127.0.0.1:8000/jobs/done" ^
  -H "Content-Type: application/json" ^
  -d "{\"device_id\":\"mill01\",\"file_name\":\"test.nc\",\"status\":\"done\"}"
```

Ответ:

```json
{
  "status": "done",
  "device_id": "mill01",
  "file_name": "test.nc"
}
```

## 5. Просмотр списка устройств

Endpoint:

```text
GET /devices
```

Этот endpoint возвращает список устройств, которые уже обращались к серверу.

Пример команды:

```bash
curl "http://127.0.0.1:8000/devices"
```

Что означает команда:

- здесь используется обычный `GET`-запрос
- `curl` просто запрашивает данные по URL

Пример ответа:

```json
[
  {
    "device_id": "mill01",
    "last_seen": "2026-03-11T22:16:23.546930+00:00",
    "status": "idle"
  }
]
```

## Порядок проверки MVP вручную

Если хотите быстро проверить, что всё работает, делайте так:

1. Запустите сервер.
2. Отправьте `POST /device/poll` от имени устройства.
3. Загрузите файл через `POST /upload`.
4. Создайте задачу через `POST /jobs`.
5. Снова вызовите `POST /device/poll` и убедитесь, что устройство получило задачу.
6. Завершите задачу через `POST /jobs/done`.
7. Проверьте список устройств через `GET /devices`.

## Важные замечания

- Регистрация устройства происходит автоматически при первом `poll`.
- Если задача уже создана, устройство получит ее при следующем `poll`.
- При выдаче задачи устройству сервер переводит статус из `pending` в `running`.
- Файл базы `wimill.db` и содержимое `storage/uploads/` не нужно коммитить в git.
- Для доступа из локальной сети сервер нужно запускать с `--host 0.0.0.0`.

## Где смотреть документацию API

После запуска откройте:

- `http://127.0.0.1:8000/docs` - если работаете на этом же компьютере
- `http://IP_ВАШЕГО_ПК:8000/docs` - если открываете с другого устройства в локальной сети
